## §1 项目概览

### 1.1 产品定位
**NovelCraft** 是一款模块化桌面小说创作平台，核心能力：
1. **富文本写作**（Tiptap 2 编辑器，支持斜杠菜单、Markdown 输入、版本历史）
2. **多 AI Agent 协作**（single / multi / swarm 三模式，可切换）
3. **Pipeline 流水线**（plan → observe → write → audit → revise → compose）
4. **Auto-pilot 自动写作**（连续生成章节，受 Quality Gate 约束）
5. **知识图谱 & 真相文件**（角色、势力、伏笔、世界观增量管理）
6. **多格式导出**（详见 §13）
7. **风格守护**（drift 监测）+ **输入治理**（input-governance 模块）

### 1.2 技术栈速查

### 1.3 模式与角色
- **single 模式**：单 agent 直接完成创作（最简，调试默认）
- **multi 模式**：多 agent 分工（plan/draft/audit/revise 各一专家）
- **swarm 模式**：蚁群式多 agent（router + supervisor + 专家池，hop 迭代）

模式切换会触发 `ai:mode:changed` 事件（详见 §8）。三模式输出质量、耗时、成本差异显著，详见 §5 测试矩阵。

## §2 环境准备与启动

### 2.1 一次性安装
```powershell
# 1. 安装 Rust（Tauri 必需，约 200MB）
winget install --id Rustlang.Rustup -e

# 2. 安装 Node.js LTS
winget install OpenJS.NodeJS.LTS

# 3. 安装 Tauri 系统依赖（Windows 自带 WebView2，无需额外）
#    macOS：xcode-select --install
#    Linux：apt install libwebkit2gtk-4.1-dev build-essential curl wget libssl-dev

# 4. 克隆/进入项目
cd C:\Users\69058\Desktop\project\自研小说软件\novel-app

# 5. 安装前端依赖
npm install
# 若失败：npm install --legacy-peer-deps
```

**预期**：`npm install` 完成无 ERR，`node_modules/` 出现，`package-lock.json` 已存在则保留。

**风险点**：
- ❌ 若代理污染 Cargo registry，需配置 `~/.cargo/config.toml` 走 RsProxy 或腾讯源
- ❌ Windows 中文用户路径（`C:\Users\<中文名>\`）可能让 Tauri 编译失败，建议复制项目到全英文路径

### 2.5 安全策略（来自 tauri.conf.json）
```

     img-src 'self' data:; connect-src 'self' https:; font-src 'self'
```
- ✅ 允许 HTTPS 出站（AI API 必需）
- ❌ 不允许 HTTP（防中间人）
- ✅ 仅允许内联样式（Tailwind JIT 需要）

### 5.5 Resilience 弹性层（resilience.ts）

#### 默认配置
```
maxRetries:  3
baseDelayMs: 1000
maxDelayMs:  30000
timeoutMs:   60000
```

#### 错误码分类
| 错误码 | 触发关键词 | 可重试 | 退避策略 |
|---|---|---|---|
| `RATE_LIMIT` | 429 / rate limit / Too Many Requests | ✅ | base × max(2^n, 4) |
| `TIMEOUT` | timeout / TIMEOUT | ✅ | 指数退避 |
| `NETWORK` | network / fetch / ECONNREFUSED | ✅ | 指数退避 |
| `AUTH` | 401 / 403 / unauthorized / invalid_api_key | ❌ | 立即放弃 |
| `SERVER` | 500 / 502 / 503 | ✅ | 指数退避 |
| `TOKEN_LIMIT` | context exceed/length/limit、token limit | ✅ | 指数退避（应触发 compressContext） |
| `UNKNOWN` | 其他 | ✅ | 指数退避 |

#### 测试用例
| 编号 | 用例 | 预期 |
|---|---|---|
| T-5.5.1 | mock 429 response | 重试，每次延迟 ≥ baseDelay × 4 |
| T-5.5.2 | mock 401 | **不重试**，立即返回空 |
| T-5.5.3 | 全部 maxRetries 用尽 | 触发 `ai:resilience:degraded` 事件，**返回空字符串**（注意：调用方需处理空字符串） |
| T-5.5.4 | timeout 超过 60s | 抛 `Timeout after 60000ms`；按 TIMEOUT 重试 |
| T-5.5.5 | 重试日志 | logger.warn 输出 `attempt N/M: CODE - msg` |
| T-5.5.6 | TOKEN_LIMIT 检测后 | 调用方应启动 `compressContext` 重试（验证集成是否到位） |
| T-5.5.7 | 已有单元测试 | `resilience.test.ts` 全部通过 |

> ⚠️ **关键风险**：`resilientCallLlm` 在全部失败后**返回空字符串**而非抛错。所有调用方（Writer/Auditor/Reviser）必须检查空字符串并向上传递错误状态，否则会出现"管线静默成功但产物为空"的 bug。

### 5.8 Workflow Engine（4 种 Pipeline 模式）

> 单例 `workflowEngine`。状态：`idle / running / paused / completed / failed`（注意与 audit-store 5 态略不同，需文档化映射）。

#### 4 种模式
| 模式 | 流程 | 触发命令 |
|---|---|---|
| `write-audit` | WRITE → 循环(AUDIT → critical=0 退出 / 否则 REVISE) | `/write` `/loop` |
| `audit-only` | AUDIT 一次 | `/audit` `/observe` |
| `revise-only` | AUDIT 找 critical → REVISE 一次 | `/revise` |
| `full-pipeline` | DirectorAgent 全流程 | `/run` `/plan` 或 multi 模式 |

#### 关键事件流（write-audit 模式）
```
pipeline:stage:start    {stage:'write'}
… [WriterAgent 流式 onDelta]
pipeline:stage:complete {stage:'write'}
pipeline:status         {status:'audit', iteration:0}
[loop]
  pipeline:quality-gate:iteration {iteration:N, criticalCount}
  pipeline:report:ready  {report, meta}
  if critical=0 → break
  if iteration>=maxRetries-1 → pipeline:quality-gate:exceeded → break
  pipeline:stage:start    {stage:'revise'}
  pipeline:status         {status:'revise', iteration}
  pipeline:stage:complete {stage:'revise'}
[end loop]
pipeline:status         {status:'completed', iteration}
```

#### 测试用例
| 编号 | 用例 | 预期 |
|---|---|---|
| T-5.8.1 | 已 running 时再次启动 | 立即返回 `{success:false, error:'Pipeline already running'}` |
| T-5.8.2 | abort() 中途 | 触发 paused 状态；下次循环检测 abortController.signal.aborted 并返回 `pausedReason:'aborted'` |
| T-5.8.3 | maxRetries=3 但 audit 一次过 | iteration=0；不进入 revise |
| T-5.8.4 | maxRetries=3 全部失败 | iteration=2 后退出（注意 ≥ maxRetries-1 检查），触发 `pipeline:quality-gate:exceeded` |
| T-5.8.5 | revise 失败（success=false） | currentContent 不变，继续下次 audit |
| T-5.8.6 | full-pipeline | 由 DirectorAgent 接管，验证完整 stage 序列 |
| T-5.8.7 | run 异常 | status='failed'，error 字段非空 |

## §6 数据层（SQLite + Drizzle）

> 本项目数据层是**双层结构**：前端 Drizzle 定义 schema 类型，但**实际持久化由 Rust 端 rusqlite 直接管理**（IPC 调用 `db::*` 命令）。Drizzle 主要用于 TypeScript 类型推导和 Schema 文档化，**不直接连 DB**。

### 6.1 数据库基本信息
| 项 | 值 |
|---|---|
| DB 引擎 | SQLite（rusqlite 0.32 + bundled） |
| 路径（Tauri） | `app_data_dir/novelcraft.db` (Win: `%APPDATA%\com.novelcraft.app\novelcraft.db`) |
| 路径（dev） | 项目根 `novelcraft.db`（drizzle.config.ts） |
| Journal mode | **WAL**（启用并发读写） |
| Foreign keys | **ON**（强制级联） |
| 自动备份 | `novelcraft.db.bak`（启动时备份） |
| 完整性检查 | 启动时 `PRAGMA integrity_check`，失败自动恢复 |
| 恢复链 | integrity_check fail → 从 .bak 恢复 → .bak 也坏则**删库重建** |

## §8 事件总线（AppEvents 全矩阵）

### 8.1 EventBus API（`core/events/index.ts`）
单例 `eventBus`，类型安全的 typed event bus。

```ts
on<K>(event, callback): () => void   // 返回 unsubscribe
once<K>(event, callback): () => void // 一次性
emit<K>(event, data): void
off<K>(event, callback?): void       // callback 缺省则清空该 event
clear(): void                        // 清空所有
```

| 编号 | 用例 | 预期 |
|---|---|---|
| T-8.1.1 | on 返回的函数调用后 | 该订阅被移除 |
| T-8.1.2 | once 触发一次后 | 自动从 listeners 中移除 |
| T-8.1.3 | 多次订阅同 event | 全部触发，按订阅顺序 |
| T-8.1.4 | callback 抛异常 | **当前实现不捕获**！会中断后续 listener（验证是否需要 try/catch 改造） |
| T-8.1.5 | emit 不存在的 event | 静默 no-op |
| T-8.1.6 | 同一 callback 重复订阅 | 触发多次（不去重）—— 验证组件 useEffect 清理避免泄漏 |

### 9.5 数据库初始化与崩溃恢复（**关键安全**）

启动流程（`lib.rs` setup）：
```
1. 创建 app_data_dir（必要）
2. 初始化 logger → app_data_dir/novelcraft.log
3. 打开 novelcraft.db
4. 设置 PRAGMA: journal_mode=WAL, foreign_keys=ON
5. 检查 PRAGMA integrity_check
   ├── 'ok' → init_db + backup_db + init_llm_tables → app.manage(DbConn)
   └── failed → drop conn → try_restore_from_backup
       ├── .bak 存在 + 整合性 ok → 替换为 .db → init_db → backup_db → manage
       └── 都失败 → delete_db_files (db/wal/shm) → 重建空库 → init_db → backup_db → manage ⚠️ 数据丢失
```

| 编号 | 用例 | 预期 |
|---|---|---|
| T-9.5.1 | 首次安装启动 | 创建空 DB；备份生成 |
| T-9.5.2 | 删除 .bak 后启动 | 启动后重新备份生成 |
| T-9.5.3 | 篡改 .db 中间数据（用十六进制工具改字节） | integrity_check 失败 → 自动从 .bak 恢复，**用户应感知到数据回滚** |
| T-9.5.4 | 删除 .db 但保留 .bak | **当前流程不会主动从 .bak 恢复**（因为 integrity_check 是基于已打开的新 DB）—— 验证此边界 |
| T-9.5.5 | .db 和 .bak 都损坏 | 删除三件套 + 重建 → 用户数据全丢失 ⚠️ **必须有显著告警** |
| T-9.5.6 | app_data_dir 无写权限 | expect 报错（panic） —— 验证是否需更友好处理 |
| T-9.5.7 | 备份策略 | 当前**只在启动时备份一次**，运行中无增量备份。建议测试关闭前再 backup |

### 9.6 Native Keychain（API key 安全）

`secret` 模块 3 个命令直连 OS 原生 keychain：
- **macOS**：Keychain Access
- **Windows**：Credential Manager（`com.novelcraft.app` 命名空间）
- **Linux**：libsecret / GNOME Keyring / KWallet

| 编号 | 用例 | 预期 |
|---|---|---|
| T-9.6.1 | set_provider_secret | 写入 OS keychain，DB 中只存哨兵（非明文） |
| T-9.6.2 | get_provider_secret | 从 keychain 读出明文 |
| T-9.6.3 | delete_provider_secret | 同时清 keychain 和 DB |
| T-9.6.4 | OS 拒绝 keychain 访问（Linux 无 keyring 服务） | Rust 应返回错误而非 panic；前端引导用户配置 |
| T-9.6.5 | 卸载 app 后 keychain 凭据 | **不会自动清理**（macOS/Win 标准行为）；建议在卸载脚本中清理 |
| T-9.6.6 | 备份/迁移到新机器 | DB 可迁移，但 keychain 数据**留在原机** —— 用户需重新输入 API key |

## §10 安全测试

> 本章测试目的：确保任何攻击者（恶意 LLM 输出、恶意插件、恶意粘贴内容、恶意 DB 文件）都不能突破安全边界获得：写文件、执行 shell、读用户数据、访问内网。
>
> 参考 OWASP Top 10 + 桌面应用专属威胁。

### 10.1 API Key 安全

> 已在 §5.2 和 §9.6 详述加密机制，本节聚焦攻击场景。

| 编号 | 用例 | 预期 |
|---|---|---|
| T-10.1.1 | 浏览器模式：localStorage 中是否能看到明文 key？ | **不能** —— 应是 AES-GCM Base64 编码 |
| T-10.1.2 | Tauri 模式：DB 中是否能看到明文 key？ | **不能** —— DB 中只存哨兵；明文在 OS keychain |
| T-10.1.3 | Tauri 模式：导出 DB 文件给他人 | 他人**无法**还原 API key（keychain 不会随 DB 走） |
| T-10.1.4 | DevTools Console 执行 `localStorage.getItem('novelcraft_providers')` | 返回加密 Base64，需密钥才能解 |
| T-10.1.5 | 内存中 API key | 调用 callLlm 时短暂存在；GC 后清理。**不应**长期持有明文 |
| T-10.1.6 | 日志中是否泄漏 key | 全部 logger 调用扫一遍：不应出现 `Bearer ` 后跟实际 key 的字串 |
| T-10.1.7 | 网络抓包 | HTTPS 出站；key 在 Authorization header 中加密传输 |

### 10.2 内容安全策略（CSP）

> tauri.conf.json 中：

## §18 故障排查 FAQ

### 18.1 启动类
| 现象 | 排查 | 修复 |
|---|---|---|
| 双击 exe 无反应 | 看任务管理器是否进程已起 | 检查 Defender / 360 拦截 |
| 启动崩溃 panic | 看 `app_data_dir/novelcraft.log` | 看是否 init_db expect 失败（权限/磁盘满） |
| 白屏 | DevTools Console 看 React 错误 | 检查 vite.config / dist 完整性 |
| `npm run tauri dev` 卡 cargo build | 是否首次编译（5-15min） | 等待或 Ctrl+C 重试 |
| 端口 5173 占用 | `netstat -ano \| findstr 5173` | 杀进程或让 Vite 自动切端口 |
| Rust 找不到 cargo | rustup 未装或 PATH 缺失 | `winget install Rustlang.Rustup` |

### 18.2 数据类
| 现象 | 排查 | 修复 |
|---|---|---|
| 启动后小说全没了 | 看 .db 是否被恢复或重建（log 中 "Integrity check failed"） | 从 `.db.bak` 手动恢复 |
| 章节内容显示乱码 | content 字段是否为合法 Tiptap JSON | 用工具校验；可能升级后 schema 不兼容 |
| 重命名角色没全部同步 | 跑 entity-rename.test.ts | 检查 §7.4 逻辑 |
| 真相文件加载失败 | logger.warn `Invalid truth file ...` | Schema 已升级，旧数据需迁移 |

### 18.3 AI 调用类
| 现象 | 排查 | 修复 |
|---|---|---|
| `/continue` 无输出 | DevTools Console 看 `ai:resilience:degraded` | API key 错 / 网络 / token 上限 |
| 流式不动 | 是否 listen 失败 | 重启 app；或换 provider |
| 输出空字符串 | resilience 静默降级（K-1） | 检查 logger error / 重新 callLlm |
| 模型路由报错 | model-router-config localStorage 是否存在 | 在 Settings 重新配置 |
| Anthropic 流式格式不对 | api_format 是否为 anthropic | Settings 中编辑 provider |
| Tauri 模式 callLlm 抛 "Tauri not available" | 是否在浏览器 dev 模式 | 用 `npm run tauri dev` |

### 18.4 编辑器类
| 现象 | 排查 | 修复 |
|---|---|---|
| 输入卡顿 | 章节字数？ | 拆分章节或检查扩展冲突 |
| 撤销丢失 | 切换章节是否清 history | Tiptap history 默认 100 步 |
| 斜杠菜单不出 | DevTools 看是否 register 了 suggestion | 检查 slash-commands/suggestion.ts |
| 中文输入冲突快捷键 | IME composition 中？ | 见 T-4.16.7 |

### 18.5 Tauri / 平台类
| 现象 | 排查 | 修复 |
|---|---|---|
| Windows Defender 拦截 | 第一次未签名 | "更多信息 → 仍要运行" 或 安装签名 |
| macOS 无法打开 | Gatekeeper 拦截 | 右键 → 打开；或公证 |
| Linux 无 dialog | libsecret/libgtk 缺失 | `apt install libwebkit2gtk-4.1-dev libgtk-3-0` |
| keyring 报错 | Linux 无 GNOME Keyring 服务 | 启动 gnome-keyring-daemon |
| 字体豆腐块 | 中文 fallback 失败 | 安装 Noto Sans CJK |

### 18.6 测试相关
| 现象 | 排查 | 修复 |
|---|---|---|
| `npm run test` 失败 | 看具体测试错误 | 修代码或更新 mock |
| Vitest 报 jsdom 错 | jsdom 版本 / DOM polyfill | npm install --legacy-peer-deps |
| coverage 0% | --coverage 配置 | vitest.config 中 coverage.provider='c8' |

### 18.7 日志位置速查
| 类型 | 位置 |
|---|---|
| App 日志 | `app_data_dir/novelcraft.log` |
| 数据库 | `app_data_dir/novelcraft.db` + .bak / .db-wal / .db-shm |
| Provider 配置（浏览器） | localStorage `novelcraft_providers` |
| 模型路由配置 | localStorage `model-router-config` |
| Bookshelf 选中状态 | localStorage `novelcraft-bookshelf` |
| Keychain（macOS） | Keychain Access → `com.novelcraft.app` |
| Keychain（Win） | Credential Manager → 通用凭据 → `com.novelcraft.app:*` |
